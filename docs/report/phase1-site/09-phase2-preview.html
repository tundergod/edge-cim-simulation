Phase 2 預覽：整合層

前面各章是元件。把它們串成端到端模擬器，需要兩個整合層模組：M3（事件引擎，處理共享頻寬競爭與重疊）與 M6（排程器，決定 op→單元映射與精度邊界）。這兩個在 Phase 1 只有 contract、沒有實作——本頁誠實預覽 Phase 2 將組裝什麼，不含任何效能數字（未實作即無可量之數）。

§0本章角色

這裡要把兩件性質完全不同的事分清楚：

① 「未實作」不是 Phase 1 的缺失。M3/M6 的行為驗證需要「先實作才能測」（toy op-stream 時序、競爭 knee、端到端 latency）。依專案規劃，實作 M3+M6 並整合正是 Phase 2 的定義內容。Phase 1 的職責只到「定義 contract + 可調參數」。

② M6 連帶一個 Phase-2 解析建模項。conversion-op（精度邊界的 dequant/requant）成本。各單元原生精度固定（CIM INT8、GPU FP16），所以邊界 cast 是 memory-bound op，由既有 M2/M4 per-op 模型 × M6 插入的邊界穿越次數計價（ADR-0004 已於 2026-06-11 修訂）。這不是測量缺口，是 Phase 2 的建模工作。

§1M3 — 事件引擎contract-only · ADR-0001

規劃要它做什麼。一個輕量 discrete-event 引擎：把 op stream 串過各單元 + 記憶體，建模 op 級並行、共享頻寬競爭的飽和 knee（~60 GB/s）、CIM ∥ GPU ∥ NPU ∥ CPU 的重疊。

Phase 1 交付了什麼。只有 contract 與可調參數；驗收門檻是 ADR-0006 的 Phase-2 系統級門檻，在 Phase 1 不 gate。per-op 延遲來源（M1、M2）已備妥餵給引擎。

狀態：未實作——行為驗證（toy op-stream 時序、~60 GB/s 競爭 knee 重現）在 Phase 2 實作後進行。

§2M6 — 排程器 / 映射器contract-only · 貢獻層

規劃要它做什麼。這是貢獻層：每個 op 決定單元（CIM/NPU/GPU/CPU）、精度、記憶體放置、dataflow，並在精度邊界插入 conversion op。

Phase 1 交付了什麼。Phase 1 的 op→單元映射是naive（固定分工：CIM 做 INT8 GEMM、GPU 做 FP16 attention）；貢獻層級的排程（依成本動態選單元、精度邊界放置）是 Phase 2 內容。Phase 1 只交付 contract 與可調參數。

狀態：未實作——conversion-op 成本是被追蹤的可調參數 + Phase-2 解析建模項（非測量缺口）。

§3conversion-op 成本Phase-2 解析建模 · ADR-0004

conversion-op 成本（precision-boundary INT8↔FP16 casts）是一個真正的 Phase-2 解析建模項，但不是量測缺口——這個區分是 Phase 1 結論的關鍵之一。

各單元原生精度固定（CIM INT8、GPU FP16，ADR-0004(c)），精度邊界由 M6 的 op→單元映射決定。邊界上的 dequant/requant 是一個 memory-bound cast（讀 int8、寫 fp16、乘 scale），其成本由既有的 M2/M4 per-op 模型 × M6 插入的邊界穿越次數解析求得——不需要專門的量測 campaign。因此「CIM-INT8 × GPU-FP16」混合精度貢獻的成本基礎 = 既有 per-op 模型 + M6 排程；Phase 2 把 cast op 插進 op stream 計價即可。

非量測缺口。精度固定於單元，故 conversion-op 不是「沒量到」的東西，而是一個 deterministic 的 memory-bound cast——由既有模型 × 邊界穿越次數解析計價。它是 Phase 2 內的建模工作，不擋 Phase 2 開工。
