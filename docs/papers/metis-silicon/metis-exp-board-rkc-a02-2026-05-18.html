03b · LLM Finding
Alpha Card LLM 硬體不相容
Device Memory Window — Alpha Card

Root Cause
Alpha 卡沒有 on-card DDR。zeDeviceGetMemoryProperties 回報的 1 GB 是 PCIe IOMMU 把 host LPDDR4 映射成 device-visible 的窗口，不是實體晶片記憶體。
所有 weight pool 在 model.json 裡都是 "memory": "ddr" → runtime 呼叫 zeMemAllocDevice，受限於此 1 GB 窗口。
Production Metis 有 4–16 GB on-card DDR，不存在此限制。

繞過路線 — 最終結論（2026-05-19 實測）
完整實作與逆向記錄：git hacks/ze_shim/（49f24fd→c103507）。LLM-on-Alpha 拆成 load 與 compute 兩段。
✅ Load 路徑 — 已從 user space 克服
ze_shim2 LD_PRELOAD，雙層攔截：

① alloc 層：zeMemAllocDevice → zeMemAllocShared（host LPDDR4，無 1GB cap）。符號須 dlopen(RTLD_NOLOAD) 解析，不能用 RTLD_NEXT（SDK dlopen 進 LOCAL namespace → 否則 segfault）。

② copy 層（關鍵）：純 alloc 攔截撞 axeShareMemoryExecute（反組譯 0x23dc0，純 PCIe-DMA executor 拒絕 shared→shared）。解法：記錄 shared pool 範圍，在 zeCommandListAppendMemoryCopy 對 pool 內 dst 直接 memcpy 並回 SUCCESS。
結果：18 pool 全分配，模型完整載入（AxInstance initialized）— 先前不可能。
❌ Compute 路徑 — closed firmware 硬牆
模型載入後 inference 立即失敗：axeLoadAndExecuteQueueElf: Wait kernel failed -1301

反組譯定論 chain：wait_kernel → wait_kernel_poll → get_kernel_sts → ctx_dev_get_sts(0xba70) → axl_read32 = 直接 PCIe 讀 AIPU device status register。

-1301 是 closed RISC-V firmware 寫進該 register 的原始值（任何 header/enum 皆查無）。AIPU 在 kernel setup 階段就 reject：.axnet queue ELF bake 的 weight 位址必須在 1GB device window 解析，host-shared 位址（0x7f…）對 compute kernel 無效。
非 timeout（KRN_TV_SEC 2→600s 無變化，立即失敗）。host 端無 lever。
根本真相：Alpha 無 on-card DDR，AIPU compute kernel 只能透過 1GB device window 定址 weights。load 是軟體限制（已克服）；compute 是 silicon/firmware 限制（user space 無解）。weights >1GB 的 LLM 在 Alpha silicon 上跑不了 compute — 這是 Alpha vs Production 的本質硬體差異。
已排除的路線
✗ 改 model.json "memory" 字段
from_json（反組譯 0x36ee0）只支援 l1/l2/ddr，無 shared/host 選項。
✗ LD_PRELOAD 攔截 get_ddr_size()
只騙過 guard check（ddr_requirement_gb），底層 zeMemAllocDevice 仍 OOM。
✗ 純 alloc 層攔截（ze_shim.c）
撞 axeShareMemoryExecute shared→shared 拒絕（已被 copy 層解決）。
✗ 調大 kernel-wait timeout
-1301 是立即 firmware reject，非逾時；600s 無效。
△ 擴大 device window（未試，高風險）
需反組譯 metis.ko；即使擴大仍須驗證 compute kernel 定址，收益不確定。

runtime dma_poll 此板固定 =1
值模式行為
0 Interrupt-driven DMA 完成後由 Metis 發送 MSI IRQ 通知 host CPU；CPU 在此期間可做其他事。此板 MSI 傳遞不可靠 → IRQ 永遠不到 → 2s 後 timeout → ZE_RESULT_EXP_ERROR_REMOTE_DEVICE → exit 134。
1 Polling ✓ 現用 CPU 忙等輪詢 DMA 狀態暫存器（tight loop）直到完成位元被置位。繞過 MSI，~0.1% CPU overhead，無 IRQ 依賴。此板唯一可用模式。
runtime dma_timeout
值行為
2（預設）2 秒後若 DMA 未完成 → axl 0000:01:00.0: DMA timeout → driver 回報錯誤。
5–10適合大模型（LLM 8B）首次載入 L2 SRAM 時間較長；避免誤判為卡死。
runtime single_msi
值模式行為
0（預設）32 MSI vectors每個 DMA channel 使用獨立的 MSI vector，最大平行度；IRQ 負載分散到 32 個 vector。
11 MSI vector所有 DMA channel 共用 1 個 IRQ line；減少 GIC 壓力，但 IRQ 序列化。dma_poll=1 時影響不大。
runtime enable_dmabuf_sync
值行為
1（預設）每次 DMA 前呼叫 dma_buf_begin_cpu_access()，DMA 後呼叫 end_cpu_access()；確保 CPU cache 與 LPDDR4 一致性（cache flush/invalidate）。
0跳過 cache sync；CPU 可能讀到 stale 舊資料，或 AIPU 讀到未 flush 的舊 weights。僅用於量測 DMA raw throughput 的 benchmark，不可用於正式推理。
compile dma_dual_channel compile_config.json
值行為
true（預設）DMA read（LPDDR4→L2，載入 activations）與 DMA write（L2→LPDDR4，輸出結果）在獨立 channel 同步進行；overlap 讀寫。
falseRead 完成後才 Write；序列化，但 DMA arbiter 壓力較低。適合調試或訊號完整性懷疑時。
Compiler / Quantization Layer
deploy.py · compile_config.json · TVM lowering · QUANTIZE_AND_LOWER mode
MEDIUM · compile-time
compile dpu_constants_home
值模式行為 & 效能影響
global.l2 On-card SRAM ✓ 現用 model load 時將 pool_l2_const.bin 整體 DMA 到 L2 SRAM；inference 中 MVM 直接讀取，單週期存取。限制：總 weights ≤ 32 MB。EfficientNet_b0 = 6.7 MB (21%)。
global.ddr Host DDR weights 留在 host LPDDR4；每個 tile 執行前才 DMA 進 L2，執行後 DMA 回（或丟棄）。無大小限制，但每 tile 多一次 PCIe round-trip；throughput 降 3–10×。適合超大模型 > 32 MB。
compile double_buffer
值模式行為
true Ping-pong ✓ 現用 分配兩組 DDR buffer（A 和 B）。AIPU 計算 buffer A 時，DMA 同步預載下一 frame 到 buffer B；完成後交換角色。DMA latency 完全隱藏在 compute 後面；代價：DDR 多佔 ~200 KB（input buffer ×2）。
false Single buffer DMA 載入 → 等待 DMA done → AIPU 計算 → DMA 輸出；完全序列化。DMA latency 直接加入 frame latency；throughput 降，但 DDR 省 ~200 KB。
compile l2_constraint
值行為
null（預設）compiler 可用全部 32 MB L2；weights 一次性全部放入。
16777216 (16 MB)強迫 compiler 在 16 MB 預算內；超出的 layer 自動 tile 成多 pass，每 pass 從 DDR 換入部分 weights。用於測試 tiling overhead 或模擬 multi-model 共用 L2 場景。
8388608 (8 MB)激進 tiling；每 pass 極小，PCIe DMA 往返次數大增，適合分析 tiling 縮放曲線。
compile pipeline_spatial_tiles / pipeline_channel_tiles
參數= true= false
spatial_tiles 允許沿 H（height）和 W（width）維度切割 feature map；每 tile 處理一個空間區域，適合大解析度輸入。 整個 H×W 必須一次放進 L1 SPM（4 MB/core）；超大 feature map 會 compile 失敗。
channel_tiles 允許沿 output channel（OC）維度切割；每 pass 計算部分 channels，結果 accumulate。 所有 output channels 必須在一個 pass 完成；需要更多 L2 空間存放 partial sums。
compile quantization_scheme
值粒度精度 vs 速度
hybrid_per_tensor
_per_channel		 Activations: per-tensor（每層 1 組 scale/zeropoint）
Weights: per-channel（每個 output channel 1 組 scale）		 精度高（接近 FP32 baseline）；calibration 較慢（需計算 per-channel min/max）。現用，EfficientNet_b0 top-1 ≈ 原始精度 -0.5%。

compile inter_operator_async
值行為
trueAIPU 執行 op N 時，DMA 預取 op N+1 的 weights 和 activations 到 L2；compute 與 data movement 重疊（software pipeline）。

compile max_tiling_attempts 現值 8
值行為
8（預設）compiler 搜尋 8 種 H-tile × OC-tile 組合，選記憶體使用率最低且 FPS 最高的方案。

locked L2 Eviction Policy
L2 SRAM 由 RISC-V 管理核的 firmware 全權管理。無任何 sysfs/ioctl/Level-Zero API 可觸發 eviction、prefetch 或 replacement policy 修改。唯一能影響 L2 內容的方式是在 compile time 透過 dpu_constants_home 決定 weights 是否進入 L2。
locked L2 Usage Monitoring
AxeDeviceMonMemPayload protobuf 的兩個欄位 reserved1 和 reserved2 均為 0（opaque，未實作）。無法從 host 查詢 L2 實際使用量或碎片狀況。唯一的 L2 使用資訊來源是 compile 時的 l2_size_used 欄位（靜態值）。
locked Hardware L2 Size
BAR2 固定映射 32 MB（0x900000000 – 0x901FFFFFF）。Alpha 版本 silicon，無法擴充。Beta/production Metis 規格未知。
locked Runtime Data Prefetch
inter_operator_async=true（compile time）是唯一可控的 prefetch 機制，且只針對 operator-level 的 DMA overlap。無法在 runtime 動態調整 prefetch 距離或策略。

Reallocation

Level-Zero 沒有 zeMemRealloc() — 必須 zeMemFree() 後重新 zeMemAllocDevice()。SDK Python 層不提供 realloc。若要動態換模型，需要：手動釋放 → 重新分配 → 重載 model。目前 load_model_instance()/unload_model_instance() API 支援此流程。

自訂動態分配演算法

SDK 層面無法直接指定 zeMemAllocShared：model.json 的 "memory" 字段經逆向確認只支援 l1/l2/ddr 三種值，沒有 shared 選項。mem_alloc_shared 函數存在但服務 I/O DMA-buf 路徑。

繞過方法（已實作驗證）：ze_shim2 LD_PRELOAD 雙層攔截可讓 LLM 模型完整載入 host LPDDR4（繞過 1GB）。但 AIPU compute kernel 無法對 host-shared weights 執行（closed firmware -1301 硬牆，見 03b）。對 vision 模型 / 自訂 allocator 實驗此技術仍有效；LLM compute 則 user space 無解。

Swap / Page Prefetch 政策

zeMemAllocDevice() 分配的 buffer 是 physically pinned — Linux kernel 不會 swap 出去（即使 system 記憶體壓力高）。可透過 mlock() 明確 pin weight files mmap 區段。madvise(MADV_WILLNEED) 可 hint kernel 預先 page-in weight 檔案（.bin files），降低首次載入 latency。DmaBufAllocator 的 Python 原始碼完整可存取（dma_buf_allocator.py），可修改 recycling pool 邏輯。

cache.h API（dcache_invalidate/writeback/flush）控制的是 RISC-V 管理核心的 64 KB SPM dcache，不是 4 MB AIPU compute buffer。

4 MB L1 是 software-managed scratchpad，沒有硬體 replacement policy。compiler 直接生成 DMA load/store 指令序列（baked into .axnet）。要修改 L1 管理策略，需要 Axelera 編譯器（closed source）。Runtime 不可修改。

L2（32 MB = 4 × 8 MB）同樣是 software-managed scratchpad，由 RISC-V firmware 管理，無 runtime eviction/prefetch API，無 sysfs/debugfs 介面。

有限的 Compile-time 旋鈕

▸ dpu_constants_home: global.l2 → weights 放 L2（預設，快）
▸ dpu_constants_home: global.ddr → weights 放 host DDR（spill，慢）
▸ l2_constraint: 16MB → 強制更緊的 L2 budget，觸發更激進 tiling
▸ Runtime 唯一可做的「swap」：zeMemFree() + zeMemAllocDevice() 手動換模型

總結：Metis 的 L1/L2 記憶體都是 software-managed scratchpad，不是 cache，沒有 replacement policy 可調。 真正的「可改性空間」在：Q1 Level-Zero C++ custom allocator + DmaBufAllocator Python， Q2 big.LITTLE scheduling（全部已驗證可寫）。 L1/L2 在 compile-time 有有限旋鈕（dpu_constants_home, l2_constraint），runtime 完全 locked。
