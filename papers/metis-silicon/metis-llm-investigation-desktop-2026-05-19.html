Metis AIPU — LLM 推論瓶頸調查報告

Host: wei-tmp-ubuntu · Ubuntu 24.04.4 / kernel 6.8.0-117 · voyager-sdk v1.6 · axelera-llm 1.6.0
主模型 Llama-3.2-1B-Instruct（4-core static, max_seq=1024）· 產出日期 2026-05-19 · 資料源 results/*.json

核心結論：Metis 在 batch-1 decode 階段是純粹 on-card DDR 權重串流受限（memory-wall）。 decode 時間 ∝ 模型位元組，跨 1B→8B 線性 r²=0.997，有效 DDR 頻寬恆定 ≈ 24.2 GB/s（CV 4.6%）。 MAC 陣列在每個 decode step ≈99% 閒置；增加核心數幾乎無幫助（4×核心僅 1.32× 加速）。

0 — Metis AIPU 架構與 host 互動（調查前提）

後續所有結論都建立在這個架構上,先講清楚。Metis 是一顆 quad-core AIPU:四個同構 AI core + 一顆 application-class RISC-V 控制核(跑 RTOS,負責 boot/周邊/編排)。AI core 採 SRAM-based 數位記憶體內運算(D-IMC):INT8 權重/激活、INT32 累加,計算直接發生在記憶體陣列內以減少資料搬移。

記憶體階層(關鍵):on-chip 容量 ≪ LLM 權重。 每 core:1 MiB 計算記憶體(D-IMC matrix-vector,tens-of-Tb/s)+ 4 MiB L1 = 5 MiB; SoC 共享 32 MiB L2;全晶片 on-chip 高速記憶體 >52 MiB。 但模型權重是 1.2 GB(1B)~7 GB(8B),放不進 52 MiB → 權重只能常駐 板載 LPDDR4x, 每步推論再串進 core。這就是 memory-wall 的架構根因:on-chip SRAM 裝不下權重,batch-1 decode 每 token 都得從 LPDDR 重新串幾乎全部權重,D-IMC 算力(53.5 TOPS/core,共 214 TOPS)被餓著 ≈99% 閒置。

E-B — 模型大小掃描・有效 DDR 頻寬・Roofline ★ headline

階段歸屬(務必看清):(a) 頻寬律量的是 DECODE 階段(自回歸每 token 的邊際 device 時間, Method-B);(b) Roofline 是 DECODE vs PREFILL 兩階段的對比。E-B 的 24 GB/s headline 結論僅針對 decode;prefill 不受此頻寬律支配(prefill 為 compute 區,見 (b))。

(a) DECODE DDR 頻寬律:Llama 族 4-core static 各尺寸,decode device-µs/token vs 權重位元組。 線性擬合 r²=0.997,斜率 → 有效 DDR BW 24.23 GB/s,CV 4.6%。 跨架構檢驗:phi3-mini(不同架構,3.61 GiB)三桶權重完全相同,decode 跨桶 ≈ 恆定 → 確認 memory-bound 同律(decode ∝ 權重、與序列長度無關);惟有效 BW ~20 GB/s 略低於 Llama 24,同 regime、常數隨架構/權重佈局微調,非精確同線。

Recommendations — 把限制用到極致(A / B1 / B2)

兩個浪費源頭、兩套對策。外部佐證:Axelera 自家 Metis M.2 Max 的官方解法就是 「記憶體頻寬 ×2 → LLM tok/s ×2」,精準命中本報告的 roofline。

A. Prefill padding 稅(今天就能做,不需廠商):

策略做法依據
① Prefill 攤提一次 prefill、KV-cache 保留、最大化每次輸出長度;多輪不重 prefill實驗 A:≥64 token 後 prefill 稅 <5%

B1. 低位元權重(量化外推,vendor 需求):現 artifact = int8(實測 1.022 bytes/param)。 套已驗證線性模型(decode ∝ weight_bytes, r²=0.997):

B2. Speculative decoding(SDK 實查:目前不可行): axelera.llm 全包無 speculative/draft/assisted API;decode 走單 token const_dpu_kernel_gen_1024 kernel,生成迴圈強制 shape (1,1),結構上無法一次驗多 token。 需 Axelera 提供 multi-token-verify 圖 + draft 整合,非現可跑實驗。

自編譯可行性(關鍵調查結論):公開 compiler ≠ LLM 編譯器。 Voyager SDK 的 axcompile(CLI/API)是 ONNX→int8→Metis 的視覺/CNN 編譯器: 輸入範例 --input-shape 1,3,224,224、用 image data 量化;ONNX 算子支援表將 attention/Matmul/Softmax 僅 whitelist 給 YOLO11,通用 LLM decoder attention/RoPE/KV-cache 不支援;compiler_configs 無任何 sequence_length/prefill/kv-cache 旋鈕; llm.md 明載「all models precompiled for now」。LLM 的雙-kernel(prefill+gen) 自回歸 artifact 只有 Axelera 內部 toolchain 能產 — 自己編 2048 桶 / int4 / batched 這條路在現有公開工具下走不通。

限制邊界(誠實標註):功耗為 nameplate-TDP 估算(PCIe Rev1 無遙測); GPU 用 ungated 模型為趨勢參考;roofline compute 天花板為 prefill 反推下界(非矽峰值); 真 roofline 轉折點需掃 batch,被 batch=1 artifact 擋死不可測。
對 Axelera 的優先需求(依天花板,因自編譯不可行故全為 vendor-only): ① batched / 動態形狀編譯圖(破 decode memory-wall,AI≈2B)· ② int4 artifact(預測 decode ×2)·③ 更短 seq 桶 256/512(短 prompt prefill ×4)· ④ Metis M.2 Max(BW×2 → tok/s×2,官方已驗)·⑤ 開放 LLM 編譯路徑。
不需廠商、立即可做:策略 ①②(prefill 攤提 + 填滿 context 預算)。

整體結論

  1. AIPU LLM 可運作，1B 模型上 ≈ 12.5× 慢於 RTX 3090。
  2. 瓶頸是 batch-1 decode 的 on-card DDR 權重串流（E1 80.7% device；其中 ~99% 為 weight I/O）。
  3. 平坦的 prefill vs 長度 = 靜態 padded graph（E3），非 GEMM 性質。
  4. decode_µs ∝ 模型位元組，r²=0.997、恆定 ~24 GB/s（CV 4.6%）— 量化證實 memory-wall。
  5. 加核心無解（1.32× / 4×）；概念解是 batching，但此 artifact 編譯為 batch=1 / static seq=1024，無法做。
  6. 限制（已誠實標註）：功耗為 nameplate-TDP 估算（PCIe Rev1 板無遙測）；GPU 用 ungated 模型，為趨勢參考非同權重對照。
